## Locked design decisions (still)

(From `docs/design.md` and the architect's Q1/Q2/Q3 responses.
Restated as anchor points for the session API design.)

- **CLI-first**: the system of record is the structured subcommand CLI.
  The session API exposes the *same* operations as inputs.
- **Three entry kinds**: `pointer | command | info`. Q2 collapse to a
  unified Node type remains a v2 candidate.
- **NodeCore substrate**: Go struct embedding + JSON Schema `$defs/node_core`
  + `allOf` mirror each other. v2 collapse is a re-tagging.
- **Concerns**: `rolodex: <uuid | null>` + `strict: bool` + `validator` +
  `depends_on: [local_ids]`. Linear dependency chains in v1.
- **Visibility, not location**: `bundled | personal | private | ephemeral`.
  Storage location is derived. Q3 collision: private > personal > bundled.
- **Storage**: JSON-per-rolodex, flat-by-uuid. SQLite FTS5 is a
  deferred rebuildable index.
- **Structured `--explore`**: `description / examples[] / notes`.
- **Verb-symmetric `activate`** with kind dispatch.
- **Paths are pure sugar over uuid** (Q3). Resolver lives in `internal/path`.

## Where to start next session — P-11.4 session API

The architect's Q1 response settled the **response envelope shape** but
it has not been implemented. The contract:

```jsonc
// Every step returns this shape, success or failure
{
  "ok": true | false,
  "session": {
    "id": "ses_...",
    "cursor": { "rolodex_id", "path", "entry_id", "mode" },
    "resolved": { "concern_local_id": "value" },
    "pending_concerns": [...],
    "version": <monotonic>
  },
  "view": { ... },        // render hint; not authoritative
  "effects": [...],       // spawn/stdout/close_session — caller executes
  "error": null | {
    "code": "INVALID_TARGET | UNRESOLVED_REQUIRED | VALIDATOR_FAILED |
             STRICT_VIOLATION | NOT_FOUND | STALE_SESSION |
             PROVIDER_FAILED | SCHEMA_ERROR",
    "message", "concern", "hint"
  }
}
```

Key principles to preserve verbatim:
- **Errors are data, not exit codes.** Reserve nonzero for protocol
  failure (bad JSON, missing session). A validation failure returns
  `ok: false` with `error.code` set.
- **Pure reducer**: same input + state → same output + effects.
  Activating a command returns `effect: spawn`; the *client* executes.
  Testability, replay, and future Unix-socket daemon all fall out of
  this.
- **`view` is a hint, `state` is the contract.** Stripping `view` and
  rendering from `state` alone must still work.
- **Session lifetime**: sliding 30-minute TTL touched on every `step`;
  explicit `dex session end`; opportunistic GC on `start`. Sessions
  are files at `~/.cache/dex/sessions/ses_*.json` with atomic rename.
  **No heartbeats** — orphaned sessions just expire.
- **Batch-friendly**: `step` should accept `{actions: [...]}` from day
  one. Per-keystroke modal dispatch is the architect's named risk;
  batch coalescing buys headroom cheaply.

CLI surface to add:
```sh
dex session start                 # → {"session_id": "ses_..."}
dex session step <id> <input>     # advance the session; emit envelope
dex session state <id>            # current envelope without advancing
dex session end <id>              # close + remove the session file
dex session list                  # debug / cleanup
```

Input action types (initial set):
- `{action: "drill", target: "<uuid|/path>"}`
- `{action: "resolve", concern: "<local_id>", value: "..."}`
- `{action: "pop"}` — up one level
- `{action: "activate"}` — emit `effect: spawn` if command,
  `effect: stdout` if info-content, drill if pointer

Architect agent from this session: id `a88229d2ed94eadbb`. Re-engage
if you want to drill into edge cases (concurrent sessions on the same
rolodex, the `view` vs `state` split during partial concern resolution,
etc.).

**Lowest-friction first move:** write the session state file format
+ a single in-memory reducer, with a test that drives a session through
drill → resolve → activate without touching the CLI dispatch yet. The
verb wiring layers on top once the reducer's shape is solid.

## Open decisions for next session

These are deferred until the session API plan lands. Each is annotated
with where it bites.

- **Session GC mechanics** (P-11.4 implementation detail): the
  architect's recommendation is opportunistic GC on `start`. Need to
  decide TTL exactly (30 min by default? configurable?) and whether
  `dex session list --stale --rm` is a manual escape hatch in v1 or
  deferred.
- **Concurrent sessions on the same rolodex**: if two sessions both
  resolve `ns=prod` then both call `activate` whose template uses `ns`,
  what's the contract? Probably "sessions are independent reducers and
  the shared state is the on-disk store, which is locked per-rolodex."
  Worth nailing in the plan.
- **`effect: spawn` envelope details**: argv, env, cwd, stdin handling.
  The current `dex activate` shells out via `sh -c`; the session API's
  `spawn` effect should probably hand back the assembled string and let
  the modal/agent decide whether to exec or display.
- **Concurrent CLI mutations during a session's lifetime**: if a user
  runs `dex rm <entry>` while a session has that entry in its cursor,
  what happens on next `step`? Probably `STALE_SESSION` with a hint to
  re-`start`. Decide at plan time.
- **`@<visibility>` anchor and `~` cursor shorthand for paths**: still
  deferred from earlier slices. Sessions are the natural home for `~`
  (current session cursor); `@<visibility>` is more about CLI ergonomics.
- **`exec` as a verb alias for `activate` on commands**: design lists
  this as a human-ergonomics nice-to-have. Stateless `dex exec` is
  trivial; session API doesn't need it.

## Known v1 trade-offs (carried forward)

These were accepted during prior slice reviews. None block the session
API but the next plan should reference them so it doesn't re-introduce
the problems:

- `path.Resolve` cycle detection is a **segment-count cap**, not a
  visited-set. Catches infinite loops, doesn't catch short cycles in
  long inputs. The session reducer should reuse `path.Resolve` directly
  (don't re-implement the walk).
- Store lookups scan every tier per call. Sessions calling `step` per
  keystroke amplify this. **Mitigation**: the reducer should cache the
  current rolodex in the session state file and only re-load on `pop`
  or explicit refresh.
- `dex search <query> --json` fails — flag must come *before*
  positional (stdlib `flag` limitation). Session API doesn't have this
  issue since steps are structured JSON; but the wider CLI library swap
  is a follow-up.
- `dex edit --content ""` produces a cryptic schema error
  (omitempty + `oneOf`). Pre-validation guards in edit.go would fix it.
- O_EXCL lockfile is stale-on-crash; error message hints at manual
  cleanup. The session API may want a staleness check (e.g., test if
  the lockfile is older than 5× the typical write window).
- `dex doctor` doesn't catch dangling refs *created by* `dex rm` because
  `pointer.to` is rolodex-id, not entry-id, in v1. Entry removal is
  graph-safe by construction. Whole-rolodex removal (deferred) would
  need a different story.
- `dex add` flag mode supports only pointer + info; commands require
  `--from-json`.
- `dex add` does not create new rolodexes — only adds entries to
  existing ones. New-rolodex creation is deferred (needs a
  slug-collision policy for top-level rolodexes).
- Concern-level mutations are full-entry replacement only (via
  `--from-json`). A focused `dex concern add/edit/rm` slice would fill
  this in.

## Addendum — session reducer + state landed

The reducer half of P-11.4 shipped on `main` via the
`2026-05-12-session-state-and-reducer` plan. `internal/session`
exposes:

- `Apply(r, st, action) → (st', envelope, err)` pure reducer
- Actions: `drill`, `pop`, `activate`, `resolve`
- Effects: `spawn` (shell_command string), `stdout`, `close_session`
- `Manager` for ses_*.json file persistence with opportunistic GC

The integration test at `internal/session/reducer_test.go::TestEndToEnd_DrillResolveActivate`
drives a session through drill → activate (stages pending) → resolve
→ activate (emits spawn). This is the exit criterion the original
handoff named.

Pinned design decisions are inlined at the top of the plan doc and
in `internal/session/doc.go`. Open follow-up:

- `dex session start|step|state|end|list` CLI verbs (next slice)
- Validator script execution + `strict` enforcement on `resolve`
- `depends_on` ordering for pending_concerns
- Drilling into a non-rolodex uuid (currently fails — caller must
  use `/path` or set the cursor directly)
- View hint population in the envelope (currently always nil)
- Round-trip coverage gap in `TestLoadRoundTripsState` (asserts only
  Resolved + Version; cursor/pending/timestamps not pinned)
- Orphan tempfile cleanup if a process dies mid-Save
- Per-session Save lockfile (single-process v1 is fine; concurrent
  processes on the same session id could race)
