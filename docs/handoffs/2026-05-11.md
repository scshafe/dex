## Locked design decisions

(From `docs/design.md`; restated here as quick reference.)

- **CLI-first**: the system of record is a structured subcommand
  CLI. The macOS modal UI is a thin client over a stateful session
  API. Agents are first-class consumers using the same primitives.
- **Three entry kinds**: `pointer | command | info`. Unified
  entry/concern Node type flagged as v2 candidate; not now.
- **Concerns**: `rolodex: <uuid | null>` + `strict: bool` +
  `validator` + `depends_on: [ids]`. Linear dependency chains in
  v1. Per-session resolved-concern cache.
- **Visibility, not location**: rolodexes carry
  `visibility: bundled | personal | private | ephemeral`. Storage
  location is derived. Agent writes default to `ephemeral`;
  explicit `dex promote` graduates them.
- **Storage**: JSON-per-rolodex, flat-by-uuid. SQLite FTS5 is a
  deferred rebuildable index, not parallel storage.
- **Structured `--explore`**: `description / examples[] / notes`.
- **Verb-symmetric `activate`** with `exec` as a command-specific
  alias.

## Open decisions for next session

These are the deferred decisions blocking implementation start.
Each is annotated with the phase that needs it:

- **Implementation language** (P-11.2 blocker): Swift (matches
  the eventual modal binary), Python (fast iteration, strong
  agent-CLI ergonomics), Go (single binary, easy distribution),
  bash (zero deps, painful at scale). Affects everything
  downstream. Worth thinking about explicitly before opening an
  editor.
- **Single-Esc-closes vs Esc-pops + double-Esc-closes** (P-11.5):
  try both in the modal; decide by feel.
- **Grid pagination vs scrolling** (P-11.6): decide when actually
  rendering >16 entries.
- **Filter scoring**: substring vs fuzzy. Substring v1; revisit if
  it feels limp.
- **Concern-provider runtime contract** (P-11.4): stdin JSON vs
  `--context` flag vs env vars. Lean toward stdin JSON.
- **Session GC** (P-11.4): TTL? Explicit cleanup? Picked when
  building the session API.
- **wm-keymap fate** (P-11.10): subsume into a dex `keybindings`
  rolodex, or keep parallel.
- **Data-structure noun**: keep "rolodex" as the noun ("a rolodex
  of entries") while the project is named `dex`, or rename it too
  so "a dex" is the noun. Cosmetic; no data-model impact.

## Where to start next session

The minimum-viable shipment per the design is **P-11.2 + P-11.4** —
stateless CLI plus session API. That unlocks the agent workflow.
The modal lands later at P-11.5 and gets polished through P-11.12.

Concrete first-step options:

1. **Decide the implementation language**, then scaffold a
   project structure (`src/`, `cli/`, `tests/`) and write the
   first stateless verb — probably `dex ls` reading an empty
   store and returning an empty array as JSON. Smallest possible
   working program, validates plumbing.

2. **Re-engage the architect** at agent id `ac322340f6090ef7a` to
   talk through:
   - the session API surface (input/output shapes for `step`,
     error model, session lifetime)
   - the entry↔concern collapse question (v1 keep separate; is
     there a refactor path that preserves data on the way to v2?)
   - language tradeoffs given the agent-first goal

3. **Pre-implementation work**: write a JSON Schema file for the
   rolodex/entry/concern shapes (validates the doc, gives editor
   support, becomes test fixture material). Could land before
   committing to a language.

4. **README polish + license**: pick MIT or Apache 2.0, add a
   LICENSE file, expand the README with a "current status" line
   and a CONTRIBUTING note (eventually).

Lowest-friction start is probably option 3 (schema) — it's
language-agnostic, codifies the contract, and is a useful
artifact regardless of which language wins.
